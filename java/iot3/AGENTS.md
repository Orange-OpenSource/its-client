# Agents Instructions ‚Äì Java IoT3 SDK

When working on files under `java/iot3/` read and follow:
- **Architectural context** (sections _Architecture Overview_ through _Key Files_): read this to understand the codebase structure before making changes.
- **Coding Conventions** (last section): rules to apply when generating or modifying any Java file in this package.

### Architecture Overview

Two main Gradle sub-modules under `java/iot3/`, each independently publishable:

| Module     | Package                   | Role                                                                               |
|------------|---------------------------|------------------------------------------------------------------------------------|
| `core`     | `com.orange.iot3core`     | Platform connectivity: MQTTv5, OpenTelemetry, LwM2M                                |
| `mobility` | `com.orange.iot3mobility` | V2X/ITS layer on top of core: CAM/DENM/CPM/MAPEM/SPATEM/MCM messaging, geo-routing |

`mobility` depends on `core` via `implementation project(':core')`. The third `examples` sub-module depends on both.

Entry points use the **inner-Builder pattern**:
```java
IoT3Mobility iot3 = new IoT3Mobility.IoT3MobilityBuilder("my-uuid", "my-context")
    .mqttParams(host, port, user, pass, tls)
    .telemetryParams(...)   // optional
    .lwm2mParams(...)       // optional
    .callback(...)
    .build();
```
A component is **only instantiated if its builder method was called** (all optional fields default to `null`).

### Build & Test Commands

Run from `java/iot3/`:
```bash
./gradlew build                  # build all modules
./gradlew :mobility:test         # run mobility tests only
./gradlew :core:test             # run core tests only
./gradlew :mobility:build --info # verbose build
```

### MQTT Topic Convention

Publish (send) and subscribe (receive) use separate prefixes:

| Direction     | Template                                                |
|---------------|---------------------------------------------------------|
| **Publish**   | `{context}/inQueue/v2x/{msgtype}/{uuid}/{quadkey_path}` |
| **Subscribe** | `{context}/outQueue/v2x/{msgtype}/+/{quadkey_path}[/#]` |

- `msgtype`: `cam`, `denm`, `cpm`, `mapem`, `spatem`, or `mcm`
- `{quadkey_path}` is generated by `QuadTileHelper.quadKeyToQuadTopic(quadkey)` which converts each char of the quadkey string into a topic level (e.g. `"0312"` ‚Üí `/0/3/1/2`)
- DENM also subscribes to a private topic `{context}/outQueue/v2x/denm/{uuid}` at init for directed hazard alerts

Subscriptions are dynamically managed by `RoIManager`: when the client calls `setRoadUserRoI(position, level, withNeighborTiles)`, `RoIManager` unsubscribes stale tiles and subscribes to new ones. Level < 22 appends `/#` wildcard.

### Multi-Version Message Support

Each message type follows the same layered structure:
```
messages/{type}/
  {Type}Helper.java           ‚Üê parse(String) / toJson(Envelope) entry point
  core/
    {Type}Codec.java          ‚Üê auto-detects version from "version" JSON field (streaming)
    {Type}Version.java        ‚Üê enum V1_x_x, V2_x_x with jsonValue()
    {Type}Exception.java
  v{version}/
    codec/   Reader + Writer  ‚Üê Jackson streaming (no object mapper / tree)
    model/   record-based POJOs
    validation/  Validator + ValidationException
```

Supported versions:

| Type   | Versions           |
|--------|--------------------|
| CAM    | `V1_1_3`, `V2_3_0` |
| DENM   | `V1_1_3`, `V2_2_0` |
| CPM    | `V1_2_1`, `V2_1_1` |
| MAPEM  | `V2_0_0`           |
| SPATEM | `V2_0_0`           |
| MCM    | `V2_0_0`           |

For instance, `CamCodec.read(String json)` returns a `CamFrame<?>` ‚Äî always check `camFrame.version()` before casting `camFrame.envelope()`.

### ETSI Unit Encoding

All positions and dynamics in messages use **ETSI-encoded integers**, not SI. Always use `EtsiConverter`:

| Physical quantity  | ETSI unit                  | Conversion                                    |
|--------------------|----------------------------|-----------------------------------------------|
| Latitude/Longitude | `int` √ó10‚Å∑ degrees         | `latitudeEtsi(deg)` / `latitudeDegrees(etsi)` |
| Speed              | `int` √ó50 (0.02 m/s steps) | `speedEtsiFromMetersPerSecond()`              |
| Heading            | `int` √ó10 (0.1¬∞ steps)     | `headingEtsiFromDegrees()`                    |
| Altitude           | `int` cm, biased           | `altitudeEtsi(meters)`                        |

ETSI time epoch starts 2004-01-01: `DELTA_1970_2004_MILLISEC = 1072915200000L`.

### Static Managers & Global State

`RoadUserManager`, `RoadHazardManager`, `RoadSensorManager`, and `ManoeuvreSessionManager` hold **static** `ArrayList` + `HashMap` state. They are initialized via `Manager.init(callback)`. There is **one instance per JVM process** ‚Äî do not run multiple `IoT3Mobility` instances in the same JVM expecting independent manager state.

### Road Object Lifetimes

Road objects are expired by a background scheduler running every second that calls `stillLiving()` on each object:

| Object                              | Lifetime           | Source                                                                                            |
|-------------------------------------|--------------------|---------------------------------------------------------------------------------------------------|
| `RoadUser`                          | 1 500 ms (fixed)   | `RoadUser.LIFETIME` constant                                                                      |
| `RoadSensor` / `SensorObject`       | 1 500 ms (fixed)   | `RoadSensor.LIFETIME` constant                                                                    |
| `RoadHazard`                        | variable           | `managementContainer.validityDuration` √ó 1 000 ms from the DENM payload                           |
| `ManoeuvreSession` (mobile source)  | 1 500 ms (rolling) | `ManoeuvreSession.MOBILE_LIFETIME_MS` ‚Äî vehicle/VRU MCM senders must emit at least once per 1.5 s |
| `ManoeuvreSession` (infrastructure) | 5 000 ms (rolling) | `ManoeuvreSession.INFRASTRUCTURE_LIFETIME_MS` ‚Äî RSU/central senders                               |

When expiry fires, the corresponding `*Expired(object)` callback is called and the object is removed from the manager's collections. CAM/CPM senders must emit at least once per 1.5 s to keep their objects alive. MCM vehicle senders must also emit at least once per 1.5 s; infrastructure senders have a 5 s window. A `ManoeuvreSession` is also removed immediately (without waiting for the timeout) when a terminating phase (TERMINATION or CANCELLATION) is received.

### TrueTime ‚Äî Accurate Timestamps

`TrueTime` is an NTP-synchronized clock initialized automatically in the `IoT3Mobility` constructor. Always use it for message timestamps:

```java
TrueTime.getAccurateTime()      // Unix ms, NTP-corrected
TrueTime.getAccurateETSITime()  // ETSI ms (Unix ‚àí DELTA_1970_2004)
```

If the NTP server (`0.pool.ntp.org` by default) is unreachable, it silently falls back to `System.currentTimeMillis()` with zero drift correction. Never call `System.currentTimeMillis()` directly for message timestamps ‚Äî it bypasses NTP correction. Use `TrueTime.setDefaultTimeServerAddress(address)` to override the NTP server.

### Model Class Pattern (Java Records + Manual Builder)

All message envelope and container POJOs are **Java records** (requires **Java 16+**). Each exposes a static `builder()` factory:

```java
CamEnvelope113 cam = CamEnvelope113.builder()
    .origin(Origin.SELF.value)
    .sourceUuid(uuid)
    .timestamp(TrueTime.getAccurateTime())
    .message(camMessage)
    .build();  // throws IllegalStateException if any field is null
```

- The `type` and `version` fields are **hardcoded in the builder's private constructor** ‚Äî do not attempt to set them; they are always set to the correct constant for that class.
- All fields are mandatory; calling `build()` with any `null` field throws `IllegalStateException("Missing field: ‚Ä¶")`.

### Road Object UUID Format

The manager key for each road object is a **composite string**, not just the `sourceUuid`:

| Object                                            | Key format                                 |
|---------------------------------------------------|--------------------------------------------|
| `RoadUser`                                        | `{sourceUuid}_{stationId}`                 |
| `RoadHazard`                                      | `{sourceUuid}_{sequence_number}`           |
| `RoadSensor` / `SensorObject`                     | `{sourceUuid}_{stationId}`                 |
| `RoadGeometry`                                    | `{sourceUuid}_{stationId}`                 |
| `SignalController`                                | `{sourceUuid}_{regionId}_{intersectionId}` |
| `ManoeuvreSession` (vehicle/VRU)                  | `{sourceUuid}_{stationId}_{manoeuvreId}`   |
| `ManoeuvreSession` (infrastructure, advised-only) | `{sourceUuid}_{stationId}_advised`         |

Two CAM messages from the same vehicle but with different `stationId` values will create two separate `RoadUser` entries.

### MCM ‚Äî Manoeuvre Coordination Message

MCM (ETSI TS 103 561) supports cooperative manoeuvre negotiation between vehicles and between vehicles and infrastructure. Only version `V2_0_0` is currently supported.

**Object model:**

| Road object        | Manager                   | Lifetime                                                                                                            |
|--------------------|---------------------------|---------------------------------------------------------------------------------------------------------------------|
| `ManoeuvreSession` | `ManoeuvreSessionManager` | Rolling ‚Äî 1 500 ms for mobile sources, 5 000 ms for infrastructure; immediate removal on TERMINATION / CANCELLATION |

**Key concepts:**

- One `ManoeuvreSession` per `(sourceUuid, stationId, manoeuvreId)` triplet for vehicle/VRU sources, keyed `{sourceUuid}_{stationId}_{manoeuvreId}`.
- Infrastructure sources that send an advised-only MCM (no `vehicleManoeuvreContainer`) are keyed `{sourceUuid}_{stationId}_advised`.
- `McmData` carries exactly one of `vehicleManoeuvreContainer` (vehicle/VRU) or `advisedManoeuvreContainer` (RSU/central advice). Use `McmData.ofVehicle(‚Ä¶)` or `McmData.ofAdvised(‚Ä¶)` factory methods accordingly.
- `ManoeuvreSession` exposes the current `ManoeuvrePhase` (derived from `McmGenericCurrentStateContainer.mcmType`), the coordination `ManoeuvreConcept` (agreement-seeking or prescriptive), the sender's `LatLng` position, the decoded planned trajectory (`List<LatLng>` waypoints of the first sub-manoeuvre), and the advice map (`Map<Long, String>` from executant station ID to advised state-change string).
- Unlike SPATEM, **MCM embeds the sender's position** in `McmMessage200.position` ‚Äî no fallback `LatLng` is needed for `sendMcm()`.
- The `generationDeltaTime` field is `(int)(TrueTime.getAccurateETSITime() % 65536)`.
- Speed in the vehicle container (`Speed.speedValue`) uses 0.02 m/s ETSI steps (identical to CAM). Speed in waypoints (`WayPoint.speed`) is an integer in **m/s** (not ETSI).

**Recommended setup:**

```java
ioT3Mobility.setManoeuvreRoI(position, level, withNeighborTiles);
ioT3Mobility.setManoeuvreCallback(new IoT3ManoeuvreCallback() {
    // mcmArrived, newManoeuvreSession, manoeuvreSessionUpdated, manoeuvreSessionExpired
});
```

**Sending MCM:**

```java
ioT3Mobility.sendMcm(mcmEnvelope200); // position is extracted from the envelope itself
```

### MAPEM / SPATEM ‚Äî Intersection & Traffic Light Model

MAPEM and SPATEM are tightly coupled: MAPEM provides static intersection geometry and SPATEM provides dynamic signal phase and timing for the same intersections.

**Object model:**

| Road object | Manager | Lifetime |
|-------------|---------|----------|
| `RoadGeometry` | `RoadGeometryManager` | Indefinite ‚Äî call `clearRoadGeometry()` when leaving an area |
| `RoadIntersection` | owned by `RoadGeometry` | Indefinite, revision-guarded updates |
| `RoadSegment` | owned by `RoadGeometry` | Indefinite, revision-guarded updates |
| `SignalController` | `SignalControllerManager` | 5 000 ms ‚Äî expires if no SPATEM update received; `signalControllerExpired()` fires on removal |
| `SignalGroup` | owned by `SignalController` | Replaced on every SPATEM update |

**Key concepts:**

- One `RoadGeometry` per MAPEM source station, keyed `{sourceUuid}_{stationId}`.
- One `SignalController` per intersection per SPATEM source, keyed `{sourceUuid}_{regionId}_{intersectionId}`.
- `RoadIntersection` and `RoadSegment` updates are revision-guarded: a new MAPEM frame is only applied when its revision is strictly higher than the stored one.
- `SignalGroup` (inside `SignalController`) exposes `SignalPhase`, `SignalColor` (including blinking state), `minEndTime`, `getPosition()` (a single representative `LatLng` stop-line position for simple map rendering), and `getLaneLevelPositions()` (a `Map<Integer, LatLng>` of all ingress lane stop-line positions keyed by lane ID, for per-lane display or V2X on-board use).
- `SignalController.getSignalGroupForLane(int laneId)` returns the `SignalGroup` controlling a specific lane (resolved from MAPEM `connects_to` data); returns `null` before MAPEM resolution or for unknown lanes.
- **Position resolution** between SPATEM and MAPEM is automatic and bidirectional: when a MAPEM arrives that matches the `regionId` + `intersectionId` of a known `SignalController`, signal group positions are resolved; and when a SPATEM arrives first, it is resolved as soon as the matching MAPEM is received.

**Recommended setup** (use the unified API rather than the separate methods):

```java
ioT3Mobility.setIntersectionRoI(position, level, withNeighborTiles); // subscribes to both MAPEM and SPATEM
ioT3Mobility.setIntersectionCallback(new IoT3IntersectionCallback() {
    // MAPEM callbacks: mapemArrived, newRoadGeometry, roadGeometryUpdated
    // SPATEM callbacks: spatemArrived, newSignalController, signalControllerUpdated
});
```

**Sending SPATEM** requires a geographic position for quadkey-based topic routing (SPATEM has no position field). The SDK resolves it automatically from the MAPEM cache; a fallback `LatLng` must be provided when MAPEM data may not yet be available:

```java
ioT3Mobility.sendSpatem(spatemEnvelope, fallbackPosition); // fallbackPosition may be null if MAPEM is guaranteed
```

### Coding Conventions

#### License Header

Every new `.java` file must start with this block (update the year range as needed):
```java
/*
 Copyright 2016-2026 Orange

 This software is distributed under the MIT license, see LICENSE.txt file for more details.

 @author Firstname LASTNAME <email@orange.com>
 */
```
When a file is fully or substantially generated by an AI agent, add a `@generated` tag immediately after `@author`:

```java
@author Firstname LASTNAME <email@orange.com>
@generated <tool> (<model>)
```
Replace <tool> with the agent tool name and <model> with the exact model name and version. Example:
```java
@generated GitHub Copilot (Claude Sonnet 4.6)
```
Do not omit either: the tool identifies the developer's workflow, the model enables future traceability if a model-specific issue is found.

#### Java Version

Java 16+ is required. Use **records** for all model/POJO classes ‚Äî never plain classes with getters/setters for message data structures.

#### Javadoc on Records

Document every record component with a `@param` tag at class level:
```java
/**
 * Short description.
 *
 * @param foo Description with unit or allowed values if relevant.
 * @param bar Optional. Description.
 */
public record MyRecord(String foo, Integer bar) {}
```

#### Builder Pattern

Builders are always a `public static final class Builder` nested inside the record. Use a private `requireNonNull` helper ‚Äî **do not** use `java.util.Objects.requireNonNull` which throws `NullPointerException` with a less descriptive message for builder misuse:
```java
public static final class Builder {
    // ...fields...
    public MyRecord build() {
        return new MyRecord(
            requireNonNull(foo, "foo"),
            requireNonNull(bar, "bar")
        );
    }
    private static <T> T requireNonNull(T value, String field) {
        if (value == null) throw new IllegalStateException("Missing field: " + field);
        return value;
    }
}
```

#### JSON / Jackson

Use **Jackson streaming API only** (`JsonParser` / `JsonGenerator`). Never use `ObjectMapper`, `@JsonProperty`, or tree model (`JsonNode`) in codec/reader/writer classes.

#### Variable Naming

Always use **full, descriptive variable names** ‚Äî never single-letter or abbreviated names. Prefer longer names over ambiguous short ones, even in loops or lambdas. Be consistent: if a type is `RoadUser`, the local variable must be `roadUser`. The pattern is the camelCase form of the type name.

#### Tests

Use **JUnit Jupiter 5** and **Mockito** only. Do not introduce AssertJ, Hamcrest, or any other assertion/mocking library.
